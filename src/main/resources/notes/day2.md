# Day 2 — prompt engineering как инженерный контракт

## Зачем я это изучаю

Prompt — это не “магическая фраза для модели”, а способ управлять качеством ответа.

Для прикладного AI engineering хороший prompt нужен, чтобы:
- уменьшить хаос в ответах
- повысить полезность результата
- задать формат
- упростить дальнейший parsing и validation

Важно:
prompt улучшает вероятность хорошего ответа, но не гарантирует строгий контракт.

---

## 1. Почему “будь умным и кратким” хуже, чем явная задача + ограничения + формат?

Потому что фраза “будь умным и кратким” слишком расплывчата.

Модель не понимает это как строгую спецификацию.  
Для нее это слабая инструкция без четкого критерия успеха.

### Проблемы такого prompt
- непонятно, что именно нужно сделать
- непонятно, насколько кратко
- непонятно, в каком виде вернуть результат
- непонятно, что считать хорошим ответом

### Лучше работает структура
Хороший prompt обычно содержит:
- задачу
- контекст
- ограничения
- формат ответа

### Пример

Плохо:
`Будь умным и кратким, проанализируй текст.`

Лучше:
`Проанализируй текст как Java backend engineer. Найди ровно 3 риска. Для каждого риска верни название, причину и одно действие. Ответ — нумерованный список без вводных фраз.`

### Вывод
Чем меньше двусмысленности, тем выше шанс получить полезный и стабильный ответ.

---

## 2. Что сильнее влияет на качество: роль, контекст, ограничения или формат?

Все 4 важны, но не одинаково.

### Практический приоритет

#### 1. Задача и ограничения
Самое важное — что именно нужно сделать и какие есть рамки.

Если задача расплывчата, роль не спасет.

#### 2. Формат ответа
Очень важен, если ответ потом читает не только человек, но и программа.

Формат снижает вариативность вывода.

#### 3. Контекст
Сильно помогает, если без него задача неоднозначна.

Контекст объясняет, зачем нужен ответ:
- для README
- для backend review
- для user-facing summary
- для decision memo

#### 4. Роль
Полезна, но обычно это усилитель, а не основа.

Фраза вроде:
`Ты senior Java backend engineer`
сама по себе не делает prompt хорошим.

### Короткий вывод
Роль полезна, но обычно сильнее влияют:
- четкая задача
- ограничения
- формат
- достаточный контекст

## 3. Когда стоит давать пример ответа?

Пример ответа стоит давать тогда, когда тебе важны:
- структура
- стиль
- степень детализации
- повторяемость формата

### Когда это особенно полезно
- модель плавает по формату
- нужен конкретный шаблон
- нужен определенный стиль ответа
- нужно уменьшить лишний текст
- нужно стабилизировать вывод на серии похожих запросов

### Когда пример особенно хорош
Если ты хочешь не просто “ответ по смыслу”, а ответ в конкретной форме, например:

- список
- таблица
- JSON-подобная структура
- блоки вида `Риск / Почему / Что сделать`

### Когда пример не обязателен
Если задача очень простая и формат не критичен.

### Риск
Если пример плохой, модель будет копировать его плохие свойства:
- лишнюю воду
- плохую структуру
- неудачный стиль

### Вывод
Пример — это способ показать модели не только *что делать*, но и *как должен выглядеть хороший результат*.

## 4. Почему prompt надо оценивать на нескольких входах, а не на одном удачном примере?

Потому что один удачный пример может создать ложное ощущение качества.

Prompt может хорошо сработать на одном тексте и плохо — на другом.

### Почему так происходит
Модель вероятностна, а входные данные бывают разными:
- короткие
- длинные
- неоднозначные
- плохо структурированные
- с недостающим контекстом

### Что бывает на практике
На одном input prompt:
- отлично держит формат
- дает четкий ответ

На другом:
- становится многословным
- теряет структуру
- повторяется
- додумывает лишнее

### Поэтому нормальная проверка — это mini-eval
Нужно прогонять prompt хотя бы на нескольких типах входов:
- простой
- средний
- неоднозначный
- “грязный” или неполный

### Вывод
Хороший prompt — это не тот, который блестяще сработал один раз, а тот, который **достаточно стабилен на серии входов**.

---

## 5. Почему prompt engineering — это не замена schema validation?

Потому что prompt влияет на вероятность, а validation проверяет факт.

### Prompt engineering
Помогает модели чаще отвечать правильно.

Но prompt не гарантирует:
- наличие всех полей
- правильные типы
- корректный enum
- валидную дату
- строгий JSON

### Schema validation
Проверяет, соответствует ли ответ контракту.

То есть:
- есть ли обязательные поля
- правильные ли типы
- допустимые ли значения
- не сломана ли структура

### Простая формула
- Prompt помогает **получить** хороший ответ
- Validation помогает **проверить**, что ответ пригоден для системы

### Backend-аналогия
Даже если внешний API “обычно возвращает нормальные данные”, ты все равно делаешь:
- DTO
- валидацию
- обработку ошибок
- защиту от некорректного payload

С LLM точно так же.

### Вывод
Prompt engineering уменьшает число ошибок.  
Schema validation защищает систему от ошибок, которые все равно останутся.

## 6. Можно ли просить AI сделать review prompt’а?

Да, это полезно.

AI может помочь:
- сократить двусмысленность
- убрать лишнее
- предложить более четкий формат
- предложить better phrasing
- найти слабые места в инструкции

### Но важно
AI-review prompt’а — это не финальная проверка качества.

Потому что модель может предложить “красивый” prompt, который:
- звучит умно
- но не дает стабильный результат на реальных inputs

### Правильный подход
Использовать AI для:
- черновика
- рефакторинга prompt’а
- поиска неясных мест
- генерации альтернативных версий

А потом проверять руками:
- на нескольких входах
- по понятным критериям
- с реальным результатом

### Формула
**AI может помочь написать prompt.  
Но качество prompt’а подтверждается только тестами.**

## Главное, что я должен запомнить

### 1.
Хороший prompt — это не “умная фраза”, а ясная инструкция.

### 2.
Самое важное:
- задача
- ограничения
- формат
- контекст

### 3.
Пример ответа полезен, когда важна стабильная форма результата.

### 4.
Нельзя оценивать prompt по одному удачному кейсу.

### 5.
Prompt не заменяет schema validation.

---

## Короткие формулы для запоминания

- **Не “красивый prompt”, а понятный контракт**
- **Task + constraints + format > vague wording**
- **Role помогает, но не спасает плохую задачу**
- **Один хороший пример ничего не доказывает**
- **Prompt улучшает вероятность, validation проверяет факт**
