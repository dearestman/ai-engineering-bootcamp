# Day 3 — structured output, schema, validation

## 1. Зачем я это изучаю

На Дне 1 я понял, что plain text — плохой контракт для приложения.  
На Дне 2 я понял, что prompt можно улучшать, но он все равно не дает жесткой гарантии формата.

Значит, следующий шаг логичный:

**модель должна возвращать не просто хороший текст, а структурированный ответ, который приложение может проверить**

---

## 2. Что такое structured output

Structured output — это подход, при котором модель возвращает ответ в заранее заданной структуре.

Обычно это:
- JSON
- JSON по схеме
- объект с фиксированными полями
- массив объектов с понятными типами

### Идея

Вместо этого:

`Вот основные риски системы: зависимость от синхронных вызовов, отсутствие идемпотентности, слабая наблюдаемость.`

мы хотим это:

{
  "risks": [
    {
      "title": "Synchronous dependency chain",
      "severity": "high",
      "reason": "Failure of downstream services breaks the whole request",
      "action": "Decouple calls via async processing"
    }
  ]
}

Почему это важно

Такой ответ:

проще парсить
проще валидировать
проще маппить в DTO
проще использовать в коде
проще передавать дальше в workflow
Короткая формула

Text удобен человеку. Structure удобна системе.

3. Зачем нужна schema

Schema — это описание того, каким должен быть ответ.

Она задает:

какие поля обязательны
какие типы допустимы
какие значения допустимы
какие лишние поля запрещены
Что schema дает practically

Schema уменьшает вариативность ответа модели.

Она делает ответ ближе к контракту, а не к “примерно правильному тексту”.

Пример

Если у меня есть поле severity, я могу явно сказать:

это строка
допустимы только:
low
medium
high

Тогда приложение не хочет видеть:

critical
normal
maybe high
очень высокий
Что важно понять

Schema не делает модель идеальной.
Она просто сильно повышает шанс получить ответ в нужной форме.

Короткая формула

Schema — это контракт формы ответа.

4. Почему schema не заменяет validation в приложении

Это одна из главных мыслей Дня 3.

Schema на стороне модели

Говорит модели:

Старайся вернуть ответ вот такой структуры

Validation на стороне приложения

Проверяет:

Я действительно получил пригодный для системы объект?

Это две разные задачи.

Что делает schema
направляет модель
ограничивает структуру
снижает хаос
помогает получить JSON нужной формы
Что делает app-side validation
проверяет DTO после парсинга
ловит пустые поля
ловит запрещенные значения
ловит бизнес-ошибки
защищает приложение от плохого ответа
Простая аналогия

Prompt и schema помогают сгенерировать хороший ответ.
Validation помогает не принять плохой ответ.

Короткая формула

Schema направляет. Validation защищает.

5. Schema validation и business validation — не одно и то же

Это очень важно.

Schema validation

Проверяет техническую корректность структуры:

есть ли поле risks
является ли оно массивом
есть ли у объекта title, severity, reason, action
допустимо ли значение severity
Business validation

Проверяет полезность для конкретной задачи:

ровно ли 3 риска
не пустой ли title
не пустой ли action
можно ли использовать ответ в сервисе дальше
Пример

JSON может быть технически корректным, но бизнесово плохим:

{
"risks": [
{
"title": "",
"severity": "high",
"reason": "something is wrong",
"action": ""
}
]
}

Формально это JSON.
Но для приложения такой ответ бесполезен.

Короткая формула

Schema отвечает: “структура верная?” Business validation отвечает: “ответ пригоден?”

6. Почему DTO лучше plain text

DTO — это уже объект, с которым приложение умеет работать нормально.

Например:

RiskReport
Risk
severity
title
action
Чем DTO лучше
поля известны заранее
типы известны заранее
удобно валидировать
удобно логировать
удобно использовать в сервисах
удобно тестировать
Plain text плох тем, что
структура нефиксированная
трудно надежно извлекать поля
приходится писать эвристики и парсеры
выше риск сломать workflow
Короткая формула

Plain text читают люди. DTO использует код.

7. Как выглядит правильный pipeline на этом этапе

Теперь мой pipeline должен выглядеть так:

prompt -> model with schema -> JSON response -> parsing -> validation -> DTO -> application logic

Что здесь важно

Модель не должна сразу считаться источником истины.
Она только генерирует payload.

Приложение:

принимает payload
парсит
валидирует
только потом использует
Это уже похоже на обычный backend

Есть внешний provider.
Есть response contract.
Есть mapping.
Есть validation.
Есть error handling.

8. Что я должен был понять на практике

После Дня 3 я должен понимать:

Хороший prompt сам по себе не заменяет structured output.
JSON лучше plain text для интеграции.
Schema помогает стабилизировать форму ответа.
Validation в приложении обязательна даже при наличии schema.
DTO — это нормальный способ перевести ответ модели в application-level объект.
9. Что особенно важно запомнить
   Prompt

Помогает модели понять задачу.

Schema

Ограничивает и структурирует ответ.

Parsing

Преобразует JSON в объекты приложения.

Validation

Проверяет, что эти объекты реально пригодны.

Итог

Только после этого ответ модели можно считать частью системы.

10. Где здесь мост к следующим темам
    Tool calling

У tool arguments тоже нужен четкий контракт.

RAG

Даже если модель получила хороший контекст, итоговый ответ все равно нужно валидировать.

Multi-agent systems

Агенты надежнее обмениваются не свободным текстом, а структурированными payload’ами.

Вывод

Structured output — это одна из базовых опор прикладного AI engineering.

11. Короткие формулы для запоминания
    Prompt помогает получить ответ
    Schema задает форму ответа
    Validation защищает приложение
    Plain text для людей, DTO для системы
    Correct JSON != useful application object

После Дня 3 я должен уметь мыслить так:
