# Day 1 — шпаргалка

## 1. LLM — это внешняя интеграция
Смотри на модель как на внешний сервис, а не как на магическую функцию.

- Controller принимает запрос
- Service решает, что делать
- Client отправляет запрос модели

Формула:
**Controller принимает → Service думает → Client отправляет**

---

## 2. Raw output != domain object
Ответ модели — это сырой результат генерации, а не готовый объект для бизнес-логики.

Почему:
- формат может плавать
- часть данных может быть неявной
- модель может ошибаться
- текст неудобно парсить

Формула:
**Raw output нужно обработать и провалидировать**

---

## 3. Timeout / Retry / Logging обязательны
LLM-вызов — обычная внешняя интеграция, значит нужны эксплуатационные механизмы.

- Timeout защищает от зависаний
- Retry помогает пережить временные сбои
- Logging помогает понять, что пошло не так

Формула:
**Timeout защищает → Retry восстанавливает → Logging объясняет**

---

## 4. Code deterministic, model probabilistic
Код приложения работает предсказуемо.  
Модель — нет.

Deterministic:
- HTTP-запрос
- сериализация
- парсинг
- валидация
- роутинг в коде

Probabilistic:
- формулировка ответа
- полнота
- точность
- соблюдение ограничений
- стабильность формата

Формула:
**Код детерминирован, модель вероятностна**

---

## 5. Plain text — плохой контракт
Текст удобен человеку, но плох для приложения.

Проблемы:
- нестабильный формат
- сложно парсить
- сложно валидировать
- неудобно использовать в workflow

Лучше:
- JSON
- schema
- validation
- typed DTO

Формула:
**Text for humans, schema for systems**

---

## Главное после Дня 1
Я должен понимать:

- LLM app = обычная интеграция
- raw output нельзя слепо использовать
- без timeout/retry/logging это не инженерный контур

- следующий шаг — structured output
