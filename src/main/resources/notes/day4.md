# Day 4 — reliability, retries, bad outputs, observability

## 1. Зачем я это изучаю

На прошлых днях я уже понял:

- как вызвать модель
- как улучшать prompt
- как получать structured output
- как валидировать ответ

Но этого еще недостаточно для нормальной интеграции.

Даже если:
- prompt хороший
- schema есть
- DTO есть

система все равно может ломаться из-за:
- сети
- таймаутов
- недоступности модели
- плохого JSON
- невалидного ответа
- ошибок в логике приложения

Значит, следующий шаг — научиться строить **устойчивый LLM pipeline**, а не просто “вызов, который иногда работает”.

---

## 2. Главная идея дня

LLM-интеграция — это обычная внешняя интеграция.

Значит, для нее нужны те же инженерные вещи, что и для любого внешнего API:

- timeout
- retry
- классификация ошибок
- logging
- observability
- fail-fast там, где retry бессмысленен

### Главная мысль

**Плохой ответ модели и сбой транспорта — это не одна и та же ошибка.**

Если не разделять типы ошибок, получится плохая система:
- retry будет запускаться там, где он не нужен
- реальные проблемы будут замаскированы
- поведение приложения станет хаотичным

---

## 3. Какие ошибки бывают

### 3.1 Transport errors
Это проблемы доставки запроса.

Примеры:
- connection refused
- timeout
- network reset
- I/O error

### Что это значит
Приложение не смогло нормально поговорить с моделью как с внешним сервисом.

### Как к этому относиться
Обычно это **retryable** ошибки.

---

### 3.2 Provider errors
Это ошибки со стороны провайдера или runtime.

Примеры:
- модель не найдена
- неверный запрос
- внутренняя ошибка сервиса
- перегрузка сервиса

### Что это значит
Запрос дошел до провайдера, но тот вернул ошибку.

### Как к этому относиться
Нужно делить на:
- **retryable**
- **non-retryable**

---

### 3.3 Parsing errors
Это ошибки, когда ответ нельзя корректно распарсить.

Примеры:
- response не JSON
- JSON битый
- структура не совпадает с ожидаемой

### Что это значит
Модель или provider вернули что-то, что приложение не может разобрать.

### Как к этому относиться
Обычно это **не retryable** ошибка.

Потому что проблема не в сети, а в содержимом ответа.

---

### 3.4 Validation errors
Это ошибки, когда JSON распарсился, но объект не проходит валидацию.

Примеры:
- пустой `title`
- `severity = critical`, когда разрешены только `low/medium/high`
- пришло 2 риска вместо 3

### Что это значит
Формально ответ получен, но он **непригоден для системы**.

### Как к этому относиться
Обычно это **не retryable** ошибка.

---

### 3.5 Business errors
Это ошибки уровня приложения.

Примеры:
- ответ технически валиден, но бесполезен
- модель не учла нужный контекст
- action слишком общий
- вывод нельзя использовать в workflow

### Что это значит
Система получила ответ, но он не решает задачу на нужном уровне качества.

### Как к этому относиться
Это уже не transport problem, а проблема **качества результата**.

---

## 4. Когда нужен retry, а когда нет

### Retry нужен, если ошибка временная

Обычно retry имеет смысл для:
- network problems
- timeout
- временной недоступности сервиса
- transient provider errors

### Retry не нужен, если ошибка логическая

Обычно retry бессмысленен для:
- model not found
- invalid request
- parse error
- validation error
- bad schema
- плохой prompt
- неправильный contract

### Почему это важно

Retry — это не “повтори все подряд”.

Retry — это политика только для тех ошибок, которые **могут исчезнуть сами при следующей попытке**.

### Короткая формула

**Retry для временных сбоев.  
Fail-fast для логических проблем.**

---

## 5. Почему fail-fast важен

Fail-fast — это подход, при котором система быстро останавливает ошибочный сценарий, а не тратит время на бессмысленные повторы.

### Где fail-fast особенно нужен
- неправильное имя модели
- невалидный input
- parse error
- validation error
- bad output format

### Почему это правильно
Если ошибка не исчезнет от повторного вызова, retry только:
- тратит время
- усложняет дебаг
- маскирует причину проблемы
- делает поведение менее предсказуемым

### Короткая формула

**Если ошибка не лечится повтором, нужен fail-fast.**

---

## 6. Почему logging и observability обязательны

Если нет логов, невозможно понять:
- где сломалось
- почему сломалось
- что именно retry’илось
- сколько длился вызов
- плохой ли был prompt
- проблема была в провайдере или в приложении

### Что стоит логировать минимум
- requestId
- model
- latency
- HTTP status
- тип ошибки
- retry attempt
- итоговый результат: success / failed

### Что логировать осторожно
- полный prompt
- чувствительные пользовательские данные
- сырой ответ модели
- большие payload’ы

### Почему это важно
LLM-интеграции тоже нуждаются в наблюдаемости, как и любой внешний dependency:
- база данных
- Kafka
- REST provider
- S3
- платежный шлюз

### Короткая формула

**Logging объясняет, observability помогает управлять системой.**

---

## 7. Как выглядит правильное мышление на этом этапе

Теперь я должен мыслить так:

`input -> prompt -> model call -> raw response -> parsing -> validation -> business usage`

И на каждом этапе возможен свой класс ошибок.

### Правильный подход
Не складывать все в одну “общую ошибку модели”.

### Неправильный подход
Считать, что любое отклонение — это просто “LLM ошиблась”.

### На самом деле
Ошибка может быть:
- в сети
- в endpoint
- в модели
- в формате ответа
- в DTO
- в validation
- в моей бизнес-логике

---

## 8. Что я должен понять после Дня 4

### 1.
Не все ошибки одинаковые.

### 2.
Retry нужен только для временных сбоев.

### 3.
Validation error — это не network error.

### 4.
Плохой output модели нельзя лечить бездумным retry.

### 5.
Logging и requestId нужны так же, как в любой внешней интеграции.

### 6.
LLM reliability — это часть backend engineering, а не отдельная магия.

---

## 9. Короткие формулы для запоминания

- **LLM integration = external dependency**
- **Not every failure should be retried**
- **Transport failure != invalid model output**
- **Retry for transient errors**
- **Fail-fast for logical errors**
- **Validation protects the application**
- **Observability makes failures understandable**

## 10. Практика на сегодня

### Цель практики
Научиться разделять ошибки и правильно на них реагировать.

---

### Шаг 1. Ввести типы ошибок в коде

Сделай минимум 4 своих exception-класса:

- `TransportException`
- `RetryableProviderException`
- `NonRetryableProviderException`
- `InvalidModelOutputException`

### Что я должен понять
Не бывает одной абстрактной “ошибки AI”.  
Нужна нормальная классификация.

---

### Шаг 2. Добавить retry только на временные сбои

Добавь retry policy только для:
- timeout
- connect error
- временных provider failures

Не запускай retry для:
- model not found
- bad request
- parse error
- validation error

### Что я должен понять
Retry — это часть reliability, а не универсальная кнопка “попробуй еще”.

---

### Шаг 3. Добавить logging

Минимум логируй:
- requestId
- model
- статус
- latency
- retry attempt
- exception type

### Что я должен понять
Без логов невозможно нормально сопровождать LLM-интеграцию.

---

### Шаг 4. Прогнать 4 сценария

#### Сценарий 1 — success
Все работает корректно.

Что проверить:
- есть ответ
- DTO валидно
- логи нормальные

---

#### Сценарий 2 — wrong model
Укажи несуществующую модель.

Что ожидается:
- fail-fast
- retry не запускается
- ошибка классифицирована как non-retryable provider error

---

#### Сценарий 3 — Ollama down / wrong port
Укажи неправильный порт или временно останови runtime.

Что ожидается:
- retry запускается
- ошибка классифицируется как transport error
- в логах видно несколько попыток

---

#### Сценарий 4 — validation failure
Сделай так, чтобы DTO стало строже, чем ответ модели.

Например:
- требуй больше полей
- запрети пустые значения
- потребуй ровно 3 объекта, а модель вернет 2

Что ожидается:
- retry не запускается
- ошибка классифицируется как invalid model output
- приложение корректно останавливает сценарий
